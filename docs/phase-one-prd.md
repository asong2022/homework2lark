# Phase 1 PRD：单题错题资产录入与教师确认

## 1. 背景

小学数学教师手中的错题材料可能来自普通拍照、高清扫描、保留学生手写与批改痕迹的原图，或去除手写内容后的空白题。扫描、增强和 OCR 只能帮助输入，不能证明题目内容正确。

本项目首先解决一个更基础的问题：怎样把教师确认的一道错题，保存成来源可追溯、机器结果与人工修订分层、状态可信、能够在未来继续复用的数学题目资产。

## 2. 用户与使用情境

- 主要用户：小学数学教师。
- 当前模式：单教师、本机个人使用。
- 典型情境：教师拿到一张作业/试卷图片，希望从中保留一道有教学价值的题，而不是立即建立学生档案或生成练习。
- 未来用户：同年级教师、学生个人、班级教学共同体；不属于 Phase 1 实现。

## 3. 用户问题

教师现有的图片、OCR 文本和手工整理结果容易混在一起：无法确认文字是否经教师确认，无法回到原图定位题目，也容易在反复识别和修改时覆盖旧数据。因此后续的再练、变式或统计没有可信的数据起点。

Phase 1 的核心验证问题是：

> 教师能否可靠地把一张作业图片中的一道题，转化为一条来源清楚、内容可修改、状态可信、未来能够继续利用的题目记录？

## 4. 目标

1. 支持上传一张合法 JPG/JPEG/PNG，并不可变保存原始文件和元数据。
2. 支持教师手动框选一道题，由后端验证坐标并生成持久化裁图。
3. 通过统一 `OCRProvider` 发起一次可追踪、可重试的 OCR 运行。
4. 同屏呈现原图位置、裁图、OCR 原文和 Provider 信息。
5. 允许教师创建不覆盖 OCR 的人工修订版本。
6. 教师确认并保存有效人工修订后，该版本立即成为当前题目版本并可进入发布门。
7. 根据 `problemId` 重新打开完整规范化记录和证据链。

## 5. 非目标

Phase 1 不实现：自动切题/整页识别、多页 PDF、图像增强/去手写、多 Provider 路由或投票、学生/班级/账号/权限、知识点与错因 AI 分析、统计看板、讲题与练习生成、图像生成/修改、RAG/向量库/知识图谱、组卷与 Word/PDF 导出、复杂队列、微服务或多租户。

这些内容只能进入 `roadmap.md` 或独立 proposal；不得为其预建大规模表和服务。

> 后续演进说明：2026-07-14 经独立 proposal 批准，仅在飞书 Base 增加一张 `错题记录` 分组表；它按同题同类错误汇总学生多选，不改变本 PRD 对本地学生账号、班级、权限和统计系统的非目标边界。

## 6. 当前工作流

```text
上传一张图片
→ 保存 SourceAsset
→ 教师框选一道题
→ 创建 ProblemRegion 并保存裁图
→ 调用当前配置的 OCRProvider
→ 保存不可变 OCRRun
→ 展示证据与 OCR 原文
→ 教师保存 ProblemRevision
→ 教师确认并保存 ProblemRevision
→ 通过 problemId 重新读取完整记录
```

## 7. Capability

### 7.1 `problem-intake`

上传、图片解码与元数据、原图读取、手动框选、坐标验证、服务端裁图、OCR 发起。

### 7.2 `problem-revision-and-confirmation`

证据展示、OCR 原文展示、人工修订版本、当前版本信息和重试反馈。

### 7.3 `mistake-record-storage`

来源、区域、OCR、修订与发布的关系持久化与规范化重建。名称中的 mistake 不表示当前实现学生错误事件。

## 8. 用户故事

- 作为教师，我希望上传原始作业图片且它不会被后续处理覆盖，以便随时核对真实证据。
- 作为教师，我希望只框选我关心的一道题，以便快速验证单题资产化流程。
- 作为教师，我希望看到裁图和 OCR 原文，以便判断机器识别是否可信。
- 作为教师，我希望修改 OCR 文本但保留原文，以便修正错误并追溯机器结果。
- 作为教师，我希望发布前系统检查当前人工版本，以免未经确认的内容进入未来题库。
- 作为教师，我希望 OCR 失败后原图和框选仍在，以便直接重试而不是重新录入。
- 作为教师，我希望用 `problemId` 重新打开题目，以便确认保存结果完整可靠。

## 9. 功能需求

### FR-01 上传与来源

- 接受单个 JPG/JPEG/PNG，默认最大 15 MiB。
- 服务端以实际解码结果验证格式，计算 SHA-256，保存宽高、大小、MIME、原文件名和 storage key。
- 原始字节不可变；宽高和裁图使用 EXIF orientation 后的显示方向，浏览器框选服务器返回的同一原图表示。
- 原图不可覆盖；hash 建索引以支持识别重复上传。
- 提供元数据与受控内容读取接口。

### FR-02 框选与裁图

- 浏览器提交 `normalized_top_left` bbox；服务端验证有限数值、0..1 范围、非零面积和最小像素尺寸。
- 后端根据原图尺寸转换为 `pixel_top_left`，生成 PNG 裁图并保存坐标和 crop key。
- 同一 SourceAsset 可以有多个区域；本次 UI 每次只录入一道题。

### FR-03 OCR

- Provider 由环境配置选择，业务层只依赖统一协议。
- 每次调用创建新的 `OCRRun`，记录 Provider/model/version、原始响应、标准文本、置信度、状态、错误码和起止时间。
- 默认使用官方托管 `PaddleOCR-VL-1.6`；本地 PaddleOCR 是可选适配器。
- 失败运行也持久化，教师可显式重试；不自动路由、降级或投票。

### FR-04 修订

- 教师可基于属于该区域的一次成功 OCR 运行保存非空修订。
- 每次保存创建新的 `ProblemRevision` 和递增版本号；旧修订与 OCR 原文不可覆盖。
- 修订可记录可选 correction note。

### FR-05 当前修订与发布门

- 每次保存修订都将其设为 `ProblemAsset.current_revision_id`，旧修订保持不可变。
- 系统不维护 `draft/needs_review/reviewed` 状态、审核时间或状态事件。
- 发布必须读取服务端当前修订，并检查非空题干、来源、裁图和 OCR 依据；缺任一项时拒绝。
- 单独重试 OCR 不改变当前人工修订。

### FR-06 规范化读取

- `GET /problems/{problemId}` 返回 source、region、当前 OCR、当前修订、全部 OCR/revision 历史、lineage、创建更新时间和发布资格；不返回审核状态历史。
- 返回原图与裁图的受控 API URL，保留原始视觉证据。

### FR-07 最小页面

- 录入页：上传、原图预览、拖拽框选、区域摘要、确认裁切、OCR 和失败重试。
- Agent 确认：原图区域、裁图、OCR 原文、Provider、修订文本和保存结果。
- 详情页：按 `problemId` 重新打开只读证据链与当前结果。

## 10. 非功能需求

- 可追溯性：所有下游记录必须保留上游 ID；机器/人工版本 append-only。
- 数据完整性：数据库事务与文件写入失败时补偿本次新增文件，不删除既有证据。
- 隐私：不要求学生身份；日志不记录图片、完整 OCR/作答文本或密钥。
- 可测试性：Provider、Storage 和 Repository 可替换；自动化默认不依赖真实模型。
- 可移植性：SQLAlchemy 查询不依赖 SQLite 专用特性，storage key 不含绝对路径。
- 可用性：错误信息说明原因与恢复动作；后续失败不清空已成功步骤。
- 可访问性：表单有标签、状态可被辅助技术读取、键盘焦点可见。
- 性能边界：单图同步处理；不引入队列。OCR 超时由 Provider 配置控制。

## 11. 风险与缓解

| 风险 | 影响 | Phase 1 缓解 |
|---|---|---|
| PaddleOCR 本地安装/模型较重 | 教师无法快速试用 | 默认托管 API；本地适配器为独立可选依赖 |
| 拍照质量或数学公式导致 OCR 不准 | OCR 文本不可信 | 强制教师确认修订；保留裁图/原图 |
| 坐标映射错误 | 裁图与选择不一致 | 统一坐标规范、服务端转换、单元/组件/E2E 测试 |
| SQLite 单写者限制 | 未来协作冲突 | 当前单教师成立；Roadmap 迁移 PostgreSQL |
| 原图含个人信息 | 隐私风险 | 本地存储/本地 OCR 默认；无学生档案；隐私安全日志 |
| 文件与数据库部分成功 | 孤儿文件或断链 | 原子写、事务和失败补偿；测试关键路径 |
| 未确认内容被发布 | 题库内容不可靠 | 后端只接受非空当前人工修订；发布门与集成测试 |

## 12. 验收边界

一次真实试用成功的定义是：教师上传一张图片，框选一道题，经一个 Provider 得到结果，确认并保存至少一个人工修订，发布后仍能按 `problemId` 重新看到原图位置、裁图、OCR 原文、修订文本与完整 lineage。

任何学生端、统计、自动切题、生成练习或多 Provider 能力均不属于本次验收。
