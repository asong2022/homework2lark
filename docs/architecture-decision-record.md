# Phase 1 Architecture Decision Record

## ADR-001：模块化产品单体与技术栈

- 状态：Accepted（2026-07-12）
- 决策：仓库采用 `apps/web`（Next.js/React/TypeScript）与 `apps/api`（FastAPI/Pydantic/SQLAlchemy/Alembic）的模块化 monorepo/product monolith。两者作为一个产品、一套数据模型和一个部署单元理解，不拆微服务。
- 理由：纵向流程需要浏览器图像交互和 Python OCR 生态；两个运行时边界清晰，同时远小于微服务基础设施。
- 代价：本地需同时运行 Node 与 Python；跨语言契约需要 OpenAPI 同步。
- 迁移：未来仍可分别部署 Web/API；当前不承诺独立团队/服务演进。

## ADR-002：Next.js App Router 与原生框选

- 状态：Accepted。
- 决策：使用 Next.js App Router、严格 TypeScript 和 DOM Pointer Events 实现矩形选择；不引入 UI、裁图或全局状态框架。
- 理由：只需要一个矩形 bbox，原生实现可控、可测试，服务端仍是裁图真值。
- 代价：需自行处理坐标缩放、反向拖拽、pointer capture、键盘替代和响应式布局。
- 风险控制：选择服务器返回的图片；前后端统一 EXIF orientation；组件和 Playwright 双层测试。

## ADR-003：同步 FastAPI + SQLAlchemy 2

- 状态：Accepted。
- 决策：使用同步 SQLAlchemy session、同步 application service 和 FastAPI 同步路由；CPU/模型调用不占异步事件循环。
- 理由：Phase 1 单教师、SQLite、同步本地文件和本地 OCR，没有异步数据库/队列的收益。
- 代价：长 OCR 占用 worker thread；本地推理超时通常不能硬中止底层计算，只能标记本次 run 超时并忽略迟到结果。
- 迁移：真实并发需求出现后再评估后台任务/进程池；这会触发 proposal。

## ADR-004：SQLite + Repository/Unit of Work 边界

- 状态：Accepted。
- 决策：开发和 MVP 使用 SQLite，开启 foreign keys；SQLAlchemy Repository 和用例级 Unit of Work 隔离业务层。
- 理由：无需外部服务，适合单教师真实试用；Repository 让未来 PostgreSQL 迁移不改变 HTTP/领域契约。
- 代价：写并发有限；SQLite timezone 和 ALTER TABLE 能力需显式处理。
- 迁移：协作/多用户阶段迁 PostgreSQL；Alembic 保持跨库类型和 batch migration 能力。

## ADR-005：规范化关系模型与 append-only 证据

- 状态：Accepted。
- 决策：关系表分别保存 source、检测证据、region、OCR run、revision、`ProblemAsset` 当前修订指针和 publication；raw response 为 JSON，图片为外部 storage key。
- 理由：直接满足数据分层、版本、失败重试和 lineage；避免把所有内容塞入可变 problem JSON。
- 代价：规范化读取需要 join/assembler；文件与数据库没有分布式事务，需补偿。
- 迁移：未来实体从 `ProblemAsset` 关联，不重构 OCR/revision 历史。

## ADR-006：本地文件存储 + StorageAdapter

- 状态：Accepted。
- 决策：原图/crop 存在 `storage/`，数据库保存不透明 key；Local adapter 做 traversal 防护、原子写、读/删/存在性。
- 理由：最快验证本地流程且不绑定云账号。
- 代价：备份、跨机器共享和容量管理需用户负责；删除必须协调数据库与文件。
- 迁移：S3/OSS/MinIO 通过同一 port；Phase 1 不实现云 Adapter 或删除 API。

## ADR-007：单 OCR Provider 配置，托管 Paddle 默认、本地 Paddle 可选

- 状态：Accepted。
- 决策：`OCR_PROVIDER` 在应用组合时只选择一个生产 Adapter。默认 `paddleocr_vl_api` 调用官方托管 Job API 与 `PaddleOCR-VL-1.6`；`paddleocr` 保留为隔离的 3.x 本地 Adapter。生产配置不暴露 Fake OCR。
- 理由：托管 VL 直接返回更适合数学公式、表格和图文页面的 Markdown，而且不需要教师电脑安装 GPU、PaddlePaddle 或 VLM 权重。自动化测试在测试目录注入 stub/transport，不把测试数据冒充产品能力。
- 运行验证：已使用用户授权的 `image2.png` 通过真实 Web/API 路径完成托管 OCRRun，Provider/model 为 `paddleocr_vl_api` / `PaddleOCR-VL-1.6`，标准化文本非空，原图与裁图仍可读。本地 Adapter 的 Windows 模式仍必须关闭 MKLDNN/oneDNN。
- 代价：托管路径需要 Token、外网和数据传输告知；结果仍是机器中间数据，原裁图和教师校对不可省略。
- 迁移：接入第二个可用 Provider 前创建 proposal；统一接口不自动引入路由/降级/投票。

## ADR-008：FastAPI OpenAPI 为跨端契约权威

- 状态：Accepted。
- 决策：FastAPI schema 生成 OpenAPI；提交 `packages/contracts/openapi.json`，使用 `openapi-typescript` 生成 Web 类型。
- 理由：避免 Python/TypeScript/文档三份手工契约漂移；前端不依赖 ORM。
- 代价：schema 变化需运行导出/生成命令，并提交生成物。
- 迁移：未来可发布 contracts package，但 Phase 1 不建独立 registry。

## ADR-009：不采用复杂基础设施

- 状态：Accepted。
- 决策：不使用 Kubernetes、微服务、Kafka、Redis、向量数据库、RAG、工作流引擎、复杂队列、多租户或完整权限平台。
- 理由：当前真实约束是单教师把一张图的一道题可靠资产化；这些设施没有解决本阶段风险，反而扩大运维面。
- 代价：同步 OCR、单机存储、无协作和无后台批处理。
- 触发条件：明确的并发、协作、批量、检索或生成需求先进入 proposal/后续 Phase。

## ADR-010：当前人工修订取代审核状态机

- 状态：Accepted（2026-07-17，取代旧审核状态事件方案）。
- 决策：`ProblemAsset` 只保存当前人工修订指针。保存非空教师确认修订后立即成为当前版本；发布检查当前修订、来源、裁图和 OCR 依据，不维护 `ReviewStatusEvent`、审核时间或复用布尔值。
- 理由：单教师 Agent 工作流中的第二次审核没有增加事实，只制造重复待办和状态漂移。教师确认仍存在，但通过不可变修订表达。
- 代价：未来若引入多人审批、责任签名或正式合规审签，需要另建协作事件模型，不能复用当前轻量假设。
- 迁移：0005 保留公开 problem ID、当前修订和 publication 远端指针，删除旧状态业务读取路径。

## ADR-011：本地证据源 + 飞书复用投影

- 状态：Accepted（2026-07-13 proposal 已批准）。
- 决策：只有存在有效当前人工修订且来源链完整的题目可显式发布；本地增加一条可重试的 publication current state，通过 `ProblemPublisher` 适配测试发布器或已认证 `lark-cli`，Base 以本地稳定 ID 幂等更新。
- 理由：飞书适合教师长期筛选、补充标签和供 Agent 选题，但不适合替代原图、OCR raw 与修订历史的证据层。
- 代价：本地与 Base 不存在分布式事务；附件追加和远端部分成功必须靠稳定 ID 查找、缺失续传和安全失败状态恢复。
- 迁移：多用户、后台同步、双向冲突合并或直接 OpenAPI 服务化必须另立任务；当前不引入队列和权限平台。

## ADR-012：举一反三采用 Skill/CLI，而非后端固定 LLM

- 状态：Accepted（2026-07-13 proposal 已批准）。
- 决策：`shi-homework2lark` 由 Codex/Hermes Agent 理解数学和生成内容，stdlib CLI 负责实时 Base schema 校验、选题、整组写回、冲突保护、单一异常标记与可用内容读取。
- 理由：AI 模型可替换，教师可在 Base 直接修订；确定性写入规则不应依赖某个模型提示词，也无需为单教师试点引入后端 LLM Provider、队列或工作流引擎。
- 代价：生成发生在 Agent 会话中，不提供后台批量任务；含图变式还要经过独立的数学题图生成、PNG 上传和图文一致性校验。
- 迁移：题图由可替换的 `wumu-jihe-html` Skill 处理，FastAPI 不绑定图片模型。未来若需要后台批量生成或多教师协作，必须另立 proposal，并继续保留预览、数学检查、异常处理和对外发布确认。

## ADR-013：双入口收集、Agent 对话优先、Base 中转核心

- 状态：Accepted（2026-07-14）。
- 决策：保留 Agent 对话与 Web 两个入口，但只维护一套 FastAPI 领域核心。日常默认从 Codex/Hermes 对话开始：整页可使用 Yescan 的 group-level 整题候选，单题截图以整图录入，教师也可直接口述学生作答和诊断；只有需要精确调整 bbox 时才打开 `/intake/{assetId}` 人工框题。全部经过 OCR 和教师确认修订后显式发布到 `错题页面/错题题目`。
- 理由：Agent 对话更适合同时吸收题图、学生真实作答、教师判断并完成后续 Base 整理；人工已经打开可视化工作台时，直接框一道题的成本低于等待和修正自动切题。产品价值在 Base 中长期复用、变式和组卷，不在扫描算法本身。
- 数据权威：Base 是教师日常中转与复用核心，显示 AI/教师整理的可读 `页面名称`/`题目名称`；SQLite/本地文件保存不可变原图、裁图、Provider raw 和修订历史，隐藏 `系统页面ID`/`系统题目ID` 连接两层。
- 代价：当前有本地证据与 Base 投影两层，需要幂等同步和失败重试；Web 自动候选兼容入口仍存在，但不作为主线。
- 迁移：Agent 层可以用 MinerU 和文档工具把 PDF/Word 页面化，再复用图片流程；FastAPI 仍不直接解析完整文档。每日一练 Word 由文档 Skill 读取内容完整且未标记异常的题目生成，不引入多 Provider 自动路由、队列或后台 LLM 服务。

## ADR-014：Base 一页一行与页面属性查找引用

- 状态：Accepted（2026-07-14）。
- 决策：`错题页面` 以物理来源图片为粒度，一个页面只保留一行；同页多道错题都通过 `所属错题页面` 关联该行。发布时先按 `系统页面ID` 幂等，未命中再按隐藏的 SHA-256 `源文件哈希` 复用唯一页面。`页码` 与 `错题来源` 只在页面表维护，题目表使用只读 lookup。
- 理由：题目数量不应复制原始页面；文件哈希能识别同一图片被重复上传产生的不同 SourceAsset ID；lookup 消除页面与题目两表重复填写造成的漂移。
- 代价：仅能自动识别字节完全一致的图片，重新压缩或裁边后的同页不会自动合并；Base 的 lookup 定义和隐藏哈希字段成为发布 schema 的必需部分。
- 风险控制：哈希多匹配时安全失败；历史归并必须先验证附件哈希、重连题目并读回双向关系，再删除重复页面行。
- 迁移：未来若需要近似图像去重，必须另立 proposal，引入感知哈希与教师确认，不能用模糊匹配静默合并教学证据。

## ADR-015：题目行人工典型错例（已取代）

- 状态：Superseded by ADR-019（2026-07-15）。
- 历史决策：早期曾在题目行维护一份可写的代表性学情备注。
- 取代原因：有了 ADR-017 的真实错题分组后，继续维护第二份可写摘要会产生事实重复、来源混淆和字段漂移；AI 模拟内容也不应占用真实作答位置。
- 迁移：真实作答只在 `错题记录` 保存，题目行改为 ADR-019 的只读聚合视图。

## ADR-016：一个对话总入口与可组合题图/文档能力

- 状态：Accepted（2026-07-14）。
- 决策：教师只需触发 `shi-homework2lark`；内部按材料调用 MinerU、PDF/doc 页面渲染、Yescan、托管 PaddleOCR-VL、`lark-base`、`wumu-jihe-html` 和文档生成能力。`wumu-jihe-html` 保持独立可复用，通过“这道题画张图”触发，输出可编辑 HTML 与 PNG。
- 理由：一个公开入口降低教师理解成本；独立能力契约避免巨型 Skill 同时耦合文档解析、OCR、飞书、几何和 Word。完整题目仍由文字、选项和视觉共同组成一个逻辑资产。
- 数据：Base 同一原题行保存编号变式、可选的分列答案解析和可选同编号题图；非空题干就是可复用变式，缺答案不构成异常。`收录序号` 使用只读自动编号稳定保存题目进入 Base 的先后顺序。Agent 区分完整裁图与 `题干图片`，文本原题默认生成纯文本变式，明确教学意图可改变表征。本地追加匿名再练事件，Base 仅投影最新摘要、掌握状态和时间。题图不是新题，反馈摘要不覆盖分组错题证据。
- 代价：跨 Skill 编排需要 Agent 正确遵守路径和确认门；附件上传不是原子 CellValue，失败时保留 `需人工处理` 并按编号重试。
- 迁移：若需要服务端自动批处理、学生身份、班级统计或后台图像模型，必须另立 proposal，不能把当前 Agent 编排直接膨胀成工作流平台。

## ADR-017：错题记录按同题同错因学生分组

- 状态：Accepted（2026-07-14）。
- 决策：飞书 Base 只新增一张 `错题记录`，不新增 `学生` 表。一行表示“一道教师确认题目 × 一次作业日期 × 一种共同错误原因 × 多名学生”；`对应学生` 使用多选，姓名、简称或匿名代号均可。先在分组级 `典型错例` 记录一到几个真实作答，再由 AI 提出并经教师确认 `错误原因`；题图通过只读引用回到原题。
- 理由：教师首先需要看同一道题有哪些真实作答、哪些学生共享同一数学原因，并据此讲评和再练。按学生逐行会重复题目和诊断；每名学生一张表更无法维护。分组行同时支持题目侧共性查看和按学生筛选个人错题集。
- 约束：同错因学生合并一行，不把名单写入 `典型错例`；主字段和系统 ID 不含姓名；稳定键使用题目、日期、分类与错因；AI 分组只作建议，写入和掌握变化必须由教师确认。`本组人数` 等于去重学生数，人数按整数显示；题目行汇总跨组学生并集，错误率保存比例并显示一位小数百分比；批改样本错误率严格大于 35% 才标记高频。
- 代价：多选学生不是规范化身份，改名、同名、跨班和原子级统计能力有限。
- 迁移：出现身份合并、班级权限或正式统计需求时另立 proposal，把多选标签迁移为规范化学生关系；当前不在 FastAPI/SQLite 增加学生域。

## ADR-018：空白模板只识别一次，批改页复用题目区域

- 状态：Accepted（2026-07-14）。
- 决策：有一份空白卷和多份同版已批改作业时，以空白卷为唯一印刷题目来源。每个空白页最多调用一次 Yescan AI Agent `scene=question-ocr` 获取结构化题目文本，再由 AI/教师对照视觉页一次性确认完整题目区域；所有学生页本地配准并复用这些区域，不重复 OCR 印刷题干。
- 理由：题目资产只需建立一次；重复识别 37 份相同印刷文本既浪费调用，也会制造 37 个不一致版本。学生页真正新增的信息是手写作答和教师批改语义。
- 身份：有班级名单时按“学号 + 姓名”核对，名单是最终真值，Base 只写姓名。没有名单时允许 AI 看图生成私有临时映射并标记置信度；必要的 Yescan 辅助只对带页序号的姓名/学号小裁图联系表调用，不拼接整页作业。
- 约束：`question-ocr` 响应没有可靠 bbox 时不得伪造坐标；红笔存在不等于答错；真实作答必须先于错因推断；低置信身份和判错进入私有例外，不静默发布。
- 迁移：跨版式、自动整卷判分、正式学生主数据和班级分析需要新 proposal；当前仍是单教师 Agent 辅助批处理。

## ADR-019：题目学情由错题记录聚合

- 状态：Accepted（2026-07-15）。
- 决策：`错题题目.典型错例` 与 `错题题目.错误表现` 改为只读 lookup，分别聚合所有关联 `错题记录` 的真实错例与可观察表现；`错题题目.错误原因` 保留文本投影，按原因附带去重学生人数。
- 理由：`错题记录` 是真实学情的唯一可写来源。lookup 能让同题多个错误组同时出现而不复制事实；带人数的原因汇总比一列无计数的原始原因 lookup 更适合教师浏览和 AI 后续选题。
- 约束：目录整理不写这三个学情列；没有真实分组时 lookup 留空，不用 AI 模拟作答填充。原因由 AI 建议，但只有教师确认的分组才进入 Base 和题目汇总。
- 迁移：保留既有题目值与关联，不重建教师手动配置的 `典型错例` lookup；新增 `错误表现` lookup 后回读多分组题目和全部常用视图。

## ADR-020：Base 采用任务视图、单一异常队列与教学仪表盘

- 状态：Accepted（2026-07-15）。
- 决策：Base 不把教师确认修订、AI 目录整理、变式检查和错题记录确认都表现为持久待办。原题在本地证据流中确认修订后发布；进入 Base 后，正常目录元数据、错题分组和题干完整的变式默认可用。答案解析是可选教师辅助，缺少它不构成异常。`错题题目.需人工处理` 只承接原题内容缺失、图文矛盾、数学冲突或身份无法匹配等真实异常；独立变式表不设复核字段，题干、唯一来源或必需题图不完整时停止写入。已确认冗余字段和变式异常视图在依赖迁移后删除。
- 视图：三张表围绕教师任务组织为 `全部页面`；`错题库/错题图片/教学优先/待生成变式/已有变式/需人工处理`；`错误分组/共性问题/按学生查询/再练跟踪`。经配置比对证明完全相同的 ID 命名重复视图可以删除。
- 仪表盘：首版只展示错题题目数、高频错题数、已有变式题数、错误人次、错误分类分布和高频错题排行。`本组人数` 跨行求和称为“错误人次”，不得误称去重学生人数；没有真实再练数据时不展示空的掌握率图。
- AI 边界：Codex/Hermes + `shi-homework2lark` 继续负责跨文件、OCR、题图、Word 和 Base 双向编排；Base AI 侧边栏用于问数和轻量分析。只有总 Skill 具备稳定 MCP/HTTP 服务入口后，才创建飞书工作流 AI Agent，避免空壳智能体。
- 代价与回滚：默认可用提高了 AI 自检质量要求，因此保留来源追溯、冲突保护、原题异常队列和对外组卷/发送确认。迁移不改业务记录；失败时按快照恢复必要结构，不恢复重复状态列作为长期规则。
