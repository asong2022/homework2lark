# 最终全自动 Skill：用户选收集方式与再练回收自动化 Proposal

## 出现的问题

现有 `shi-homework2lark` 已能收题、整理 Base、生成变式和练习纸，但入口会由 AI 自行判断处理模式，且批改后的练习扫描件只能由教师/AI 手工描述反馈，不能批量回到 manifest 和 Base。这与用户要求交付给 Hermes 的“最终全自动版”不一致。

## 为什么超出原范围

原 capability 明确把“返回扫描件自动定位、批改结果提取和批量反馈写回”留在后续任务。新增该能力会扩展再练反馈的数据流、幂等事件和 Skill 入口状态，因此必须先记录 proposal，不能只改提示词声称已完成。

## 建议方案

1. 收集开始前固定由用户选择：教师精选、匿名批改统计、实名绑定统计；AI 不代选。
2. 用户选择后由单一 Agent 工作流连续调度 OCR、教师批量审核、Base、变式、组卷和反馈。
3. 新增确定性 `workflow.py` 固化选择，新建 `retry_batch.py` 将 AI/OCR 观察映射到不可变练习 manifest。
4. 再练先生成只读计划，再追加本地事件并更新 Base 当前投影；不确定项进入异常清单。
5. 一次正确不自动等同“已掌握”，红笔存在不自动等同“答错”。

## 对数据模型的影响

- 不新增学生表或学生账号。
- 练习 manifest 继续冻结 `Rxx → questionId/variantId`。
- 新增本地 append-only 再练批次事件；Base 只保存当前教师工作台投影。
- 实名模式只在私有名单/manifest 中保存学号，Base 默认仍写姓名。

## 对技术栈的影响

- 仅新增 Python CLI、reference 和测试，不新增服务、队列、向量库或数据库。
- 看图继续由 Hermes/Codex 或 `shi-ocr` 完成；确定性脚本不内置大模型 Provider。
- Base 写入继续委托 `lark-base/lark-cli --as user`，保持单一认证面。

## 实施成本

中等：需要入口状态、manifest 定位、事件幂等、批次异常隔离、文档/评测和 Hermes 真机验证。无需修改 FastAPI 数据库或 Web 前端。

## 是否建议进入当前阶段

建议。用户于 2026-07-15 明确要求交付给 Hermes 的最终全自动版，并明确决定“收集方式由用户选择，不由 AI 判断”，视为本 proposal 的产品授权；仍保留教师审核、第三方上传和 Base 写入门。
