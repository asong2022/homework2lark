# Proposal：学生个人精准练习

> 状态：已批准实施。用户于 2026-07-15 明确要求开发学生个人精准练习功能。

## 出现的问题

现有 `错题记录.对应学生` 已能按姓名筛出个人错题，但正式练习纸仍是全班共用版本：姓名、学号留空，所有学生使用同一组题。这样没有把已经积累的个人错误证据转化为真正轻量、可打印的个别化练习。

## 为什么超出原范围

原 `agent-assisted-learning-loop` 只批准全班共用练习，并明确把学生工作流留到后续阶段。个人练习新增了班级名单读取、按学生筛选、个别化选题、私有身份清单和每名学生不同 Word，因此需要单独记录边界，不能悄悄把它塞进全班模板。

## 建议方案

只在 `shi-homework2lark` Skill/CLI 增加一个教师侧个人练习编排器：

```text
私有班级名单中的学号
→ 在本地解析姓名与匿名实例号
→ Base 按 错题记录.对应学生 精确筛选
→ 同一原题的多条错因记录聚合
→ 未掌握优先、近期优先、同层级兼顾错误分类
→ 每道来源错题先取一道原题
→ 题量仍有余量时按来源轮转补充可用变式
→ 生成个人 manifest
→ 生成已填写姓名、学号的个人练习纸 Word
```

- 默认目标 6 题，允许教师或 Agent 指定 1～12 题；证据不足时宁可少出，不用无关题凑数。
- 默认排除 `已掌握`；`需再练 → 练习中 → 未开始` 依次优先，同一层级按最近作业/再练时间排序，并在不降低掌握优先级的前提下兼顾错误分类。
- 每个来源错题先选一题，避免同一薄弱点挤占整张练习；仍有容量时再按来源轮转加入可用变式。
- 生成前保留教师/Agent 可调整题量和选题的预览；算法是可解释默认值，不是自动掌握判断。

## 对数据模型的影响

- 不新建 `学生` 表、班级表、账号、权限或本地学生数据库。
- 不修改飞书 Base 的四张业务表，也不写回任何选题状态。
- 班级名单是本地私有输入，只用于“学号 + 姓名”核对；Base 查询仍使用已验证姓名。
- 个人 manifest 在本地保存姓名、学号和批次内匿名实例号 `S001...`。打印页码使用 `YYYYMMDD-NN-Sxxx-P页码`，不编码姓名或学号。
- 每个 `Rxx` 继续保存精确 `questionId` 或 `variantId`，为回收练习时定位个人、题目和来源预留入口。

## 对技术栈的影响

- 新增一个纯 Python Skill CLI，复用现有 `lark-cli` 只读 Base 网关和 `practice_sheet.py` Word 生成器。
- 首版曾扩展正式练习纸 manifest 为 `personal-practice-v1`；真实样张反馈后，新生成改用扁平 `personal-practice-v2` 并让 Word 自然分页，V1 仅兼容读取。
- 不修改 Next.js、FastAPI、SQLite，不新增后台队列或 LLM Provider。

## 隐私与失败边界

- 真实名单和个人 manifest 必须存放在教师私有、Git 忽略目录；标准输出和普通错误日志不打印姓名、学号或完整题干。
- 同名学生因 Base 当前只保存姓名而无法可靠区分时停止生成，不猜测、不合并。
- 学生没有未掌握错题时返回可解释的空结果，不自动从全班高频题补题。
- Base、题图附件或来源关系不完整时跳过异常题并报告数量；原题不使用 `需人工处理=true` 的记录，变式题按非空题干、唯一来源关系及已回读的可选题图判断可用性。

## 实施成本与建议阶段

成本中等，主要是个人选题契约、Base 只读适配、Word 身份版式和隐私测试。建议作为 Phase 1C / 单教师个人精准练习试点实施；批量生成全班、扫描回收自动判分、学生历史统计和账号系统继续留到后续 proposal。
